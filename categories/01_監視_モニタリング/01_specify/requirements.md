# 監視・モニタリング 要件定義（アセスメント仕様）

> 自社監視基準に照らして、既存 Azure 環境が網羅されているかをアセスメントする

## 1. 背景と目的

### 1.1 背景

- 自社では監視基準を定めており、運用対象の Azure 環境がその基準に適合していることを確認する必要がある
- 既存環境には、監視が有効化されている領域と未整備の領域が混在している可能性がある
- 監視の有無を仕様化することで、是正の優先度を判断できるようにする

### 1.2 目的

- 既存 Azure 環境が自社監視基準を満たしているかを判定する
- 未適合または部分適合の監視項目を明確化する
- 是正の判断材料となるアセスメント結果を残す

## 2. 監視基準の対象

自社監視基準は、以下の 11 項目で構成する。各監視基準は、対象リソースの種類に応じて適用可否を判定する。

| No. | 監視基準名 | 何を確認するか |
|-----|-----------|--------------|
| M-01 | 稼働監視 | リソースやサービスが継続稼働しているか |
| M-02 | CPU 使用率監視 | CPU の高負荷状態を把握できるか |
| M-03 | メモリ使用率監視 | メモリ逼迫を把握できるか |
| M-04 | ログ監視（イベントログ・Syslog） | OS ログを収集・確認できるか |
| M-05 | プロセス監視 | 重要プロセスの稼働・停止を確認できるか |
| M-06 | ディスク空き容量監視 | ディスク枯渇の兆候を把握できるか |
| M-07 | レスポンス監視 | 特定アドレス・ポートの応答確認ができるか |
| M-08 | ネットワーク監視（ピアリング間正常性） | ピアリングや経路の正常性を確認できるか |
| M-09 | ファイアーウォール監視（許可・拒否） | 許可・拒否トラフィックを確認できるか |
| M-10 | バックアップステータス監視 | バックアップの成功・失敗を確認できるか |
| M-11 | リソース正常性監視 | Azure プラットフォーム起因の異常を確認できるか |

## 3. アセスメント対象スコープ

### 3.1 対象範囲

アセスメント対象は、[Account/accounts.yml](../../Account/accounts.yml) に定義されたアカウント配下の以下の範囲とする。

- `subscriptions[].subscription_id` が示すサブスクリプション
- `subscriptions[].resource_groups` に列挙されたリソースグループ
- それらの範囲に存在する全 Azure リソース

### 3.2 対象外

- 開発・テスト環境（dev/sandbox）サブスクリプション
- Azure DevOps / GitHub Actions パイプラインリソース
- サードパーティ SaaS サービスへの外形監視

## 4. アセスメント結果の扱い

### 4.1 判定区分

| 判定 | 定義 |
|------|------|
| ✅ 適合 | 対象の監視基準を満たしている |
| ⚠️ 部分適合 | 一部は満たすが不足がある |
| ❌ 非適合 | 監視が未整備または確認できない |
| N/A | 対象リソースに適用しない |

### 4.2 出力物

| 成果物 | 内容 |
|--------|------|
| アセスメントシート | 対象リソースごとの判定結果 |
| ギャップ一覧 | 未適合・部分適合の詳細 |
| 是正タスク | 是正が必要な項目の管理 |

### 4.3 アセスメント結果の必須項目

各対象リソースについて、少なくとも以下を出力する。

| 項目 | 内容 |
|------|------|
| 対象情報 | サブスクリプション、リソースグループ、リソース名、リソース種別 |
| 現状の Azure サービス | すでに利用している監視系サービス・設定 |
| 適用監視基準 | M-01〜M-11 のうち該当する項目 |
| 判定 | 適合 / 部分適合 / 非適合 / N/A |
| 不足内容 | 何が足りないか |
| 必要な導入サービス | 不足を補うために追加すべき Azure サービス |
| 証跡 | 判定根拠となる設定やクエリ結果 |

## 6. Markdown記述・パワーポイント出力指針

### 6.1 Markdownアウトプットの記載ルール

- 各監視基準ごとに「判定（適合／部分適合／非適合）」を必ず明記する
- 部分適合・非適合の場合は、
  - 「完全適合にするために必要なこと（ToDo）」
  - 「なぜそれが必要か（理由）」
  - 「参考になる公式ドキュメントやナレッジサイトのURL」
  を必ず記載する
- 判定根拠となる証跡（KQL クエリ例、CLI / PowerShell コマンド、API 応答等）も明記する

### 6.2 パワーポイント化を意識したシナリオ構成

- 各監視基準ごとに「1シート」想定でMarkdownセクションを作成する
- 最終結果レポートでは、ID略記（例: M-01、S1-T01）のみで記載しない
- セクション見出し例：
  - `## 稼働監視の適合性判定`
  - `## CPU使用率監視の適合性判定`
- 各セクション内で以下の順序で記載する：
  1. 判定（適合／部分適合／非適合）
  2. 判定理由・現状
  3. 完全適合にするために必要なこと（ToDo）
  4. なぜ必要か（理由）
  5. 参考サイトURL
  6. 判定根拠となる証跡（KQL例など）

#### サンプルテンプレート

```
## 稼働監視の適合性判定

- 判定: ⚠️ 部分適合
- 判定理由: Heartbeatは取得できているが、Resource Healthの監視設定が未実施
- 完全適合にするために必要なこと: Resource Healthアラートの有効化
- なぜ必要か: プラットフォーム障害時の早期検知・通知のため
- 参考サイト: https://learn.microsoft.com/ja-jp/azure/service-health/resource-health-overview
- 判定根拠（KQL例）: `Heartbeat | where Computer == "<VM名>" | where TimeGenerated > ago(1h)`
```

- このテンプレートに従い、各監視基準ごとにMarkdownを記述すること
- 最終的なパワーポイント出力時は、各セクションが1シートに対応する

### 6.3 最終結果ファイルの配置ルール

- 最終結果ファイルは、監視モニタリングカテゴリ直下の `output/` フォルダに配置する
- 最終結果ファイル名は `final-assessment-result.md` を標準とする
- 一時的な検証用ファイルは `05_verify/` に置いてよいが、最終版は必ず `output/` 配下へ集約する

### 6.4 タスクID表記ルール

- 最終結果ファイルでは、タスクIDのみの記載を避ける
- タスクIDを記載する場合は、必ず「そのタスクで何を実施するか」の説明文を併記する
- タスクIDを知らない読者でも理解できるよう、業務観点の自然文で記載する
- 監視基準ID（M-xx）についても同様に、IDのみの記載を避け、監視観点名を本文に記載する
