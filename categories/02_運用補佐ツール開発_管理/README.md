# カテゴリ 02: 運用補佐ツール開発・管理

## 概要
このカテゴリでは、インフラ業務の自動化・効率化を目的とした、以下の成果物を対象とします：

- **Web UI/Web アプリケーション**（React, Vue, Angular など）
- **自動化スクリプト**（Python, PowerShell, Bash など）
- **バッチ処理**（定期実行ツール、データ処理パイプライン）
- **テストツール**（スクリプト・自動テストフレームワーク）
- **運用ワークフロー**（Logic Apps, Azure Functions, Jenkins など）

## SDD 7工程の実行順序（必須）

このカテゴリで取り扱うすべての依頼は、以下の工程順序を **厳密に守る** 必要があります：


## 工程別ファイル配置

| 工程 | フォルダ | ファイル | 説明 |
|------|---------|---------|------|
| 1 | 01_specify | `<request-folder>/requirements.md` | 要件定義書（What/Why のみ） |
| 2 | 02_plan | `<request-folder>/plan.md` | 実装設計書（技術選定、How） |
| 3 | 03_tasks | `<request-folder>/tasks.md` | タスク分解（粒度: 30分以内） |
| 4 | 04_implement | `<request-folder>/implement.md` + `<request-folder>/app/` または `<request-folder>/scripts/` | 実装記録 + 実装コード |
| 5 | 05_verify | `<request-folder>/verification.md` | テスト計画・実行ログ |
| 6 | 06_migration | `<request-folder>/migration.md` | 本番展開手順 |
| 7 | output | `<request-folder>/result.md` | 最終成果物 |

## このカテゴリに適用される品質ゲート

1. **要件品質ゲート** （STEP 1 完了時）
   - スキル: `sdd-requirements-quality-gate`
   - 確認項目: What/Why 分離、受入条件の測定可能性、制約整合

2. **仕様-設計整合ゲート** （STEP 2 完了時）
   - スキル: `sdd-spec-plan-alignment`
   - 確認項目: 要件と設計の対応関係、出力パス整合

3. **検証証跡ゲート** （STEP 5 完了時）
   - スキル: `sdd-verify-evidence-recorder`
   - 確認項目: テスト設計の受入条件対応、テスト実行ログの具体性

## 禁止事項（必読）

### ❌ 実装ファーストアプローチ
```
❌ 間違ったパターン:
   1. ヒアリングを受ける
   2. 直接 app/ や scripts/ フォルダでコード実装
   3. 後付けで requirements.md を作成

✅ 正しいパターン:
   1. ヒアリングを受ける
   2. requirements.md（What/Why）を仕様品質ゲート合格まで作成
   3. plan.md（技術選定）を作成
   4. tasks.md（タスク分解）を作成
   5. それから初めて app/ や scripts/ でコード実装
```

### ❌ コードの不正配置
```
❌ 禁止:
   - 01_specify/<request-folder>/app/  ← ここに実装コードを置かない
   - 02_plan/<request-folder>/scripts/  ← ここに実装コードを置かない

✅ 正しい配置:
   - 04_implement/<request-folder>/app/
   - 04_implement/<request-folder>/scripts/
```

### ❌ 仕様なしでの設計・実装決定
```
❌ Specify 段階でやってはいけないこと:
   - React + Vite を選定（これは PLAN で決定）
   - API エンドポイント設計（これは PLAN で決定）
   - DB テーブル定義（これは PLAN で決定）
   - 実装コード生成（これは IMPLEMENT で実施）
```

## よくある質問

### Q1: 仕様がまだ不確実ですが、試行錯誤で app フォルダを作っても良いですか？
**A**: NO。必ず requirements.md → plan.md → tasks.md の順で進めてください。試行錯誤は tasks.md の段階で「プロトタイプ検証」として記述し、それに基づいて実装タスクを修正してから進めます。

### Q2: 急いでいるので、STEP 3（tasks.md）を省略できますか？
**A**: NO。STEP 3 がタスク粒度を明確にするので、実装フェーズで効率が大幅に向上します。省略は後で工数オーバーランを招きます。

### Q3: 実装中に仕様が変わったら、どうすればいいですか？
**A**: 
1. 変更を 01_specify に反映
2. 仕様品質ゲート再実行
3. plan.md, tasks.md を更新
4. implement.md に「変更理由」を記録

### Q4: 複数の request-folder を同時に進められますか？
**A**: はい。ただし、各 request-folder ごとに STEP 1 → STEP 2 → ... の順序を守る必要があります。
