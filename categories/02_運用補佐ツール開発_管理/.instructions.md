# カテゴリ 02: 運用補佐ツール開発・管理 - 仕様優先実行モード

## 概要
このカテゴリで扱うすべての依頼（Web UI、スクリプト、自動化ツール、バッチ処理など）について、**仕様駆動開発（SDD）の7工程を厳密に守る**ことを目的とします。

## 強制される実行順序

### ✅ 許可される工程順序
```
1️⃣  ヒアリング（8項目）
        ↓
2️⃣  カテゴリ判定: 02_運用補佐ツール開発・管理 と確定
        ↓
3️⃣  01_specify/<request-folder>/requirements.md 作成
     （What/Why のみ。How は一切含めない）
        ↓
4️⃣  仕様品質ゲート実行
     スキル: sdd-requirements-quality-gate
     判定: 合格 を確認してから次へ
        ↓
5️⃣  02_plan/<request-folder>/plan.md 作成
     （技術スタック、実装方針を記述）
        ↓
6️⃣  03_tasks/<request-folder>/tasks.md 作成
     （30分以内の粒度でタスク分解）
        ↓
7️⃣  04_implement/<request-folder>/implement.md + コード実装
     （ここで初めてコード生成・実装を開始）
        ↓
8️⃣  05_verify/<request-folder>/verification.md 作成
     （テスト設計・検証実行）
        ↓
9️⃣  06_migration/<request-folder>/migration.md 作成
     （本番展開手順）
        ↓
🔟 output/<request-folder>/result.md 作成
     （最終成果物）
```

### ❌ 禁止される実行パターン
```
❌ パターン1: ヒアリング → 直接 04_implement へ着手
❌ パターン2: requirements.md 作成前に app/ フォルダでコード実装
❌ パターン3: plan.md 完成前に コード生成 
❌ パターン4: 品質ゲート不合格のまま次工程へ進む
❌ パターン5: 実装ファイル（app/, scripts/）を 04_implement 外に配置
```

## 実装コード生成時の制約

### コード生成を開始できる条件
以下をすべて満たすまで、コード生成（app/, scripts/）を開始しない：

- [ ] `01_specify/<request-folder>/requirements.md` が存在し、What/Why が完成している
- [ ] 仕様品質ゲート（`sdd-requirements-quality-gate`）で「合格」判定を得ている
- [ ] `02_plan/<request-folder>/plan.md` が完成し、技術スタックが確定している
- [ ] `03_tasks/<request-folder>/tasks.md` が完成し、タスク分解が確認されている
- [ ] ユーザーが明示的に「実装開始」を指示している

### コード配置ルール
```
✅ 正しい配置:
   categories/02_運用補佐ツール開発_管理/04_implement/<request-folder>/app/
   categories/02_運用補佐ツール開発_管理/04_implement/<request-folder>/scripts/

❌ 間違った配置:
   categories/02_運用補佐ツール開発_管理/01_specify/<request-folder>/app/  ← 絶対禁止
   categories/02_運用補佐ツール開発_管理/02_plan/<request-folder>/app/     ← 絶対禁止
   categories/02_運用補佐ツール開発_管理/03_tasks/<request-folder>/app/    ← 絶対禁止
```

## 各工程のチェックリスト

### STEP 1: Specify（01_specify）
- [ ] What: 実装対象が具体的に記述されている（What のみ）
- [ ] Why: ビジネス効果・必要性が記述されている
- [ ] 受入条件: 測定可能な項目がリストアップされている
- [ ] 制約: Constitution.md との整合が取れている
- [ ] 出力先: 7工程すべての出力先が明記されている

**このステップ完了後、必ず仕様品質ゲートを実行する**

### STEP 2: Plan（02_plan）
- [ ] 技術スタック: 言語、フレームワーク、DB が決定している
- [ ] アーキテクチャ: システム構成図が記述されている
- [ ] データモデル: エンティティ定義、API 仕様が明記されている
- [ ] 実装手順: 工程を 5-7 ステップに分割している
- [ ] リスク・ロールバック方針が記述されている

**このステップ完了後、STEP 3 へ進む**

### STEP 3: Tasks（03_tasks）
- [ ] タスク粒度: 1 タスク = 30 分以内の工量
- [ ] 依存関係: タスク間の先後関係が明示されている
- [ ] 優先度: 実装順序が記述されている
- [ ] 所有者: 各タスクの担当者が明記されている

**このステップ完了後、実装開始を許可する**

### STEP 4-5: Implement + Verify（04_implement, 05_verify）
- [ ] アプリ / スクリプトが要件を満たしている
- [ ] テストが受入条件に対応している
- [ ] テスト実行ログが 05_verify に記録されている

### STEP 6-7: Migration + Output（06_migration, output）
- [ ] 本番デプロイ手順が記述されている
- [ ] 最終成果物が output に配置されている

## 仕様変更が発生した場合

**ビジネス要件の変更や重大なバグ発見時**は以下の手順を取る：

1. `01_specify/<request-folder>/requirements.md` を更新
2. 仕様品質ゲートを再実行
3. `02_plan/<request-folder>/plan.md` を更新
4. 必要に応じて `03_tasks/<request-folder>/tasks.md` を更新
5. その後、実装の変更部分を `04_implement/<request-folder>/implement.md` に記録

**逆方向（実装 → 仕様）の変更は原則禁止**

## 品質ゲート実行時期

| 工程 | ゲート種別 | 実行タイミング | スキル |
|------|---------|-------------|--------|
| Specify | 要件品質ゲート | requirements.md 完成後 | `sdd-requirements-quality-gate` |
| Specify-Plan | 整合ゲート | plan.md 完成後 | `sdd-spec-plan-alignment` |
| Verify | 証跡ゲート | verification.md 完成後 | `sdd-verify-evidence-recorder` |

## このカテゴリのエージェント

## エラーが検出された場合

このカテゴリの `.instructions.md` ルールに違反した場合（例：requirements.md なしでコード生成を試みた場合）：

1. 違反の原因を特定する
2. 仕様工程（STEP 1-3）に戻る
3. 先行工程を完成させる
4. 改めて実装フェーズに進む
